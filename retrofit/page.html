What is responsive design?

A design that will respond regardless of screen size, platform and orientation
i.e. it should work on all things - monitors, smartphones, tablets, tvs, etc

Why do responsive design?

Why bother?

  • Modern mobile devices are designed to display non-responsive websites effectively
  • Users are used to seeing their favourite sites on a desktop so will expect the same on their mobile device - don't like change!
  • Takes more developer time/effort/cost

Pros

  • Usability - make it as simple and easy as possible for the user to view content with minimal interaction
  • Google recommends it - having your desktop and mobile site on a single URL makes it easier 'for Google’s algorithms to assign the indexing properties to your content'
    • In the 'old' days having a mobile strategy was to have a separate site for mobile (e.g. m.nhs.uk) and for desktop. Having one single site means Google only has to index the one site.
  • Some sites don't look good when zoomed in - difficulties with scrolling (Google maps ), lots of scrolling with drop down menus

NHS Choices pre responsification

The iPhone automatically rescales site to fit the screen, allows user to double tap, pinch and zoom

NHS Choices mega menu!

The mega menu on our pregnancy and baby guide fills the whole screen and is a bit overwhelming. Need to do lots of scrolling and is a poor user experience.

NHS Choices map view - map trap

If you zoom in on the page and then scroll over to view the map, if the map then fills the whole screen, you can't scroll to any other part of the page - you would be trapped in scrolling around on the map.

Why responsify NHS Choices?

The stats (End of Nov 2013 and Mid Jan 2014, Source: Webtrends)

  • 50.5 million visits to NHS Choices
  • 22 million from a mobile phone (44% approx.) of which
  • 15.9 million were to treatment and conditions (31% of all visits) of which
  • 9.3 million from an iPhone (18.4% of all visits)

That's a lot.

10% time

Last year, NHS Choices introduced the idea of 10% time.

Concept used at Google (they use 20%) whereby you would do your job for 80% of the time and the remaining 20% of the time you can work on anything else

As long as it advances Google in some way.

Gmail, Google Talk and Adsense all started off as 20% time projects.

My 10% time

NHS Choices 10% time involved presenting an idea to the Choices 'dragons' for approval.

If successful, you could work on your project on a Friday afternoon for a set period of time

At the end of the time period, you would present your idea to all.

Based on our stats, my idea was to...

"Optimise Health A-Z for small viewports"

Why Treatments and Conditions (Health A-Z)?
Because this is the most visited area of the site so will affect the most number of users, hopefully for the better.

The solution

For a 'quick win', I worked out that I could make changes to just 2 files to optimise the whole of Health A-Z

  • 1 javascript file (choices.js)
  • 1 css file (healthaz.css)

Even better, these files can be patched easily without the need to do a build.

I didn't want to change any page templates which would be risky, nor did I want any changes to be made in content by the content team as that could potentially mean changing every page. There are lots of them.

How it was done (part 1)

Inject meta viewport element in the head using javascript

<meta name=“viewport” content=“width=device-width, initial-scale=1.0” />

What is this?

  • It was developed by Apple for their mobile Safari to control layout on mobile browsers
  • iOS Safari and other mobile browsers have a viewport of 980px - the width of the viewing area on the screen
  • When a webpage loads, it squeezes the page to fit inside the viewport
  • The meta viewport tag allows you to override this
  • So, content="width=device-width" sets the viewport width to be equal to the width of the device's screen
  • On an iPhone 4, the width is 320px in portrait mode, and 480px in landscape mode

  • This allows us to use media queries and target specific widths

Inject meta viewport in to the head

Javascript:
$('head').append('<meta name="viewport" content="width=device-width, initial-scale=1.0" />');

How it was done (part 2)

Use media queries to target smaller viewports and override/create new styling

@media (max-width:640px) {
  ...
}

What is this?

  • It is a W3C recommended standard and was added in the CSS3 specification
  • Allows you to target specific viewport widths, as well as other things such as height, orientation, aspect ratio, resolution

New homepage carousel code

Create a new carousel for mobile, based on the existing HTML.

function carouselResponsive() {
  var $active = $('.features .feature-current');
  
  if ( $active.length == 0 ) $active = $('.features div.feature-wrap:last');

  var $next =  $active.next().length ? $active.next()
  : $('.features div.feature-wrap:first');

  $next.css({opacity: 0.0})
  $(".features").css("height", $next.height());
      
  $next.addClass('feature-current').animate({opacity: 1.0}, 400, function() {
    $active.removeClass('feature-current');
  });
}

Back to top link

Inject back to top link as user starts to scroll. Considered useful if the page has lots of content, which most of our pages do.

$(window).scroll(function() {
  if ($(this).scrollTop() > offset) {
    $('.back-to-top').fadeIn(duration);
  } else {
    $('.back-to-top').fadeOut(duration);
  }
});

$('.back-to-top').click(function(event) {
  event.preventDefault();
  $('html, body').animate({scrollTop: 0}, 500);
  return false;
})

Caveats

It's not perfect.

It should only be regarded as a short term fix because it is not the most efficient and can be done much better from the ground up.

It doesn't respond well if you continuously resize your window on the desktop because of the javascript that runs at load time.

It is a hack but it does seem to work and hopefully will benefit the increasing number of mobile users.
